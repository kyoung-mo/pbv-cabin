## 요약 (Executive Summary)

본 프로젝트는 자율주행·PBV 시대의 가변 실내(reconfigurable cabin)를 A2 스케일 모형으로 구현한다. 핵심은 **사용자 입력이 액추에이터에 직접 도달하지 못하도록**, 입력과 액추에이션 사이에 결정적 안전 검증 계층(슈퍼바이저 인터록)을 두는 것이다.

좌석 구성은 짐칸(CARGO) 모드와 발판(램프) 축을 제외하고, 차박 모드를 **Full-space**로 재정의(뒷좌석을 앞으로 붙이고 시트를 접어 뒷공간을 비움)했으며, **휴식(Rest)** 모드를 신설했다. 발판 제거로 재배치 축은 **8축**이다(4 리클라인 + 2 앞좌석 회전 + 2 뒷좌석 슬라이드).

슈퍼바이저 플랫폼은 **라즈베리파이5(RPi5) + MCP2515 CAN 모듈**로 단일화한다. RPi5 한 노드가 입력 수용·인터록 판단·안무 지휘·**통합 제어 화면**(좌석 상태·기어·인터록·로그 + 좌석 제어 + **실시간 3D 디지털 트윈**)을 모두 담당하고, CAN 접속은 SPI로 연결한 MCP2515 모듈이 맡는다.

> **🔧 구현 현황 한 줄 요약** — 입력(레이싱휠 USB-HID + 터치) → 슈퍼바이저 인터록(기어=P 전이잠금, DRIVE 모드에서만 D/R) → CAN(500kbps, 표준 11비트 ID) → 3개 존 ECU(프론트 F103 / 리어 F446 / 구동 F446) 액추에이션까지 **단방향 파이프라인이 실제로 동작**한다. 반면 설계가 강조한 **상위 안전계층(Heartbeat 생존감시, 독립 감시 노드, SafeAbort 송신, IWDG)은 아직 코드에 없다.** E-stop 채널(CAN 0x10)은 3개 ECU가 **수신**은 구현했으나 **슈퍼바이저가 송신하지 않는다**(현재는 테스트 스크립트로만 발동). 즉 "정상 동작 파이프라인"은 완성 단계, "고장 대응 안전망"은 설계 단계다.

이 구조는 실차 E/E 패턴(도메인 컨트롤러 + 액추에이터 ECU + E-Gas 다단 감시)에 대응하며, 분산 안전제어·드라이브-바이-와이어를 하나의 캡스톤으로 통합 실증한다.

## 1. 개요 & 포지셔닝

### 1.1 세 개의 층

1. **입력층** — 터치 HMI(모드 선택 / 좌석 개별 제어) + 아우라 레이싱휠(조향각·기어 D/P/R·페달). 두 입력 모두 동일한 슈퍼바이저로 수렴한다.
2. **안전 슈퍼바이저층** — 입력을 사전조건 검증 후 승인/거부하고, 모드 전이 안무를 지휘한다.
3. **CAN 액추에이션층** — STM32 존 ECU가 모터 제어 + 로컬 안전 반응을 수행한다.

### 1.2 핵심 차별화

- **확률적 입력과 결정적 안전 경로의 분리** — 어떤 입력도 액추에이터를 직접 제어하지 않는다. 모든 입력은 "요청"이며, 안전 인터록 통과 후에만 CAN 명령으로 변환된다. (🔧 슈퍼바이저 `_apply_gear`·`_seat_cmd_approved` 게이트로 구현)
- **분산 ECU 아키텍처** — 슈퍼바이저(RPi5) + STM32 존 ECU ×3을 CAN으로 묶은 존(zone) 기반 구조. (🔧 4노드 실장. 설계의 5번째 "독립 감시 노드"는 미실장 — §14)
- **독립 2차 안전 감시자** — 📐 슈퍼바이저 하트비트를 감시하다 끊기면 SafeAbort만 발동. (🔧 **미구현** — Heartbeat 메시지 자체가 어떤 노드에서도 송수신되지 않음)
- **기능안전 지향 설계** — 주행 금지 인터록, 순차 안무, 앤티핀치. (🔧 주행금지·순차안무·앤티핀치(리어)만 구현. IWDG·페일세이프 타임아웃 일부만/미구현 — §6·§14)
- **수동 드라이브-바이-와이어** — 레이싱휠 페달 → 토크 요청 → CAN → 구동 ECU. (🔧 페달은 **RPi5가 USB-HID로 읽어** Drive_Cmd로 변환. 구동 ECU는 CAN 슬레이브 — APPS ADC 아님, §5.2)

### 1.3 설계 대원칙 (엄수)

| 원칙 | 내용 | 🔧 구현 |
| --- | --- | --- |
| 입력 분리 | 입력은 요청일 뿐, 인터록 통과 후에만 CAN 명령이 된다. | ✅ 슈퍼바이저 게이트로 구현 |
| 분산 (조율 vs 반사) | 느린 조율(모드 전이)은 슈퍼바이저, 빠른 반사(주행 차단·끼임 정지)는 로컬 STM32. | ◐ 주행차단(구동)·끼임(리어)만 로컬 구현 |
| 결정은 한 곳 | 동작 명령 권한은 슈퍼바이저 하나. 감시 노드는 정지만. | ✅ 동작명령은 슈퍼바이저만 송신 (감시노드 자체가 미실장) |
| 고장 대응 = fail-safe | 고장 시 안전정지. | ◐ 구동 ECU 300ms 명령손실 정지만. 하트비트 기반 정지 미구현 |

> **포지셔닝 주의** — 본 캡스톤은 Edge AI 과정 산출물이나 카메라/AI 인지를 포함하지 않는다. "분산 ECU·기능안전·실차 네트워크·드라이브-바이-와이어 아키텍처에 집중한 선택"임을 선제 설명한다.

## 2. 좌석 구성 & 모드 정의

### 2.1 좌석 (🔧 코드 확정)

`SEAT_LABELS = driver, passenger, rear_left, rear_right` (vehicle_state.py). 앞좌석 2 + 뒷좌석 2.

| 좌석 | 위치 | 축 |
| --- | --- | --- |
| 운전석(driver) | 앞 좌 | 리클라인 + 회전 |
| 조수석(passenger) | 앞 우 | 리클라인 + 회전 |
| 뒷자리1(rear_left) | 운전석 뒤 | 리클라인 + 슬라이드 |
| 뒷자리2(rear_right) | 조수석 뒤 | 리클라인 + 슬라이드 |

### 2.2 모드 (전부 HMI 선택) — 🔧 코드상 문자열, enum 아님

코드에서는 한글 문자열로 관리한다(`MODE_PRESETS`, `ModeSelect.qml`). Python enum 클래스는 없고 문자열 일치로 판별.

| 모드(설계) | 🔧 코드 문자열 | 앞좌석 | 뒷좌석 | 용도 |
| --- | --- | --- | --- | --- |
| DRIVE | `"주행"` (`DRIVE_MODE`) | 전방·정위치(회전0°) | 직립·정위치 | 정상 주행. **유일하게 기어 D/R 해제** |
| CONFERENCE | `"회의"` | 180° 후방 회전 | 직립·정위치 | 전·후석 마주봄 |
| FULL-SPACE | `"Full-space"` | 정위치 | 폴딩 + 앞으로 슬라이드 | 뒷공간 비워 평탄 적재(차박) |
| REST | `"휴식"` | 정위치 | 리클라인 + 뒤로 슬라이드 | 누워서 휴식 |

> 🔧 `"주행"` 모드만 특수: 기어 D/R 잠금 해제 + 좌석 정위치 복귀("settle") 후 자동 AMBIENT 전환 로직(`_check_drive_settle`) 보유.

### 2.3 진입 트리거 & 기어 인터록 (🔧 구현)

- **HMI 모드 선택** — DRIVE/CONFERENCE/FULL-SPACE/REST → 해당 모드 안무.
- **HMI 좌석 개별 제어** — 좌석 하나를 골라 축을 직접 조그(SeatDetail).
- **레이싱휠 주행** — DRIVE 모드 + 기어=D + 주행-가능 시에만 구동.

기어 D/P/R 자체가 인터록 입력이다. **CAN 인코딩: P=0, D=1, R=2** (`GEAR_CAN_VALUE`; UI 슬라이더 표기 순서 R-P-D와는 별개).

- 모드 전이(좌석 안무)는 **기어=P(0)일 때만 허용** (🔧 `selectMode`/`_seat_cmd_approved`가 D/R 중 차단).
- 주행/후진은 **모드=DRIVE일 때만** D/R 진입 허용 (🔧 `_apply_gear`가 비-DRIVE에서 D/R 거부).
- D↔R 급전환 차단(반드시 P 경유) (🔧 `requestGearIndex` 한 칸 제한).

## 3. 모션 / 액추에이터 정의 — 📐 설계 vs 🔧 실측

> **⚠️ 원안과 가장 크게 갈린 절.** 설계는 "스텝 전부 TMC2209 + NEMA17 통일"이었으나, 실제 구현은 존마다 다른 드라이버·모터를 쓴다. 발표/BOM은 아래 🔧 실측 기준으로 맞출 것.

| 좌석/부위 | 움직임 | 📐 설계 부품/드라이버 | 🔧 실제 구현 (코드) | 개수 |
| --- | --- | --- | --- | --- |
| 앞좌석(2) | 180° 회전 | NEMA17 + TMC2209 | **28BYJ-48 + ULN2003** (GPIO 하프스텝 비트뱅잉, `step.c` 8상 테이블) | 2 |
| 전 좌석(4) | 리클라인 | 아두이노 서보 | **SG90 + TIM2 PWM** (50Hz, 500–2500µs→0–180°) | 4 |
| 뒷좌석(2) | 슬라이드 | NEMA17 리드스크류 + TMC2209 | **NEMA17 리드스크류 + TMC2208** (STEP/DIR/EN, `STEPS_PER_MM≈396.8`, 스트로크 100mm) | 2 |
| 주행 바퀴 | 전후진 | JGB37-520 + L298N | **JGB37-520 + L298N, 2륜 차동** (개루프, 엔코더는 텔레메트리만) | 2 |

**재배치 8축** = 회전 2 + 리클라인 4 + 슬라이드 2. **+ 주행 서브시스템** = 레이싱휠 페달 → 구동 바퀴 2륜.

### 좌석별 축 매트릭스 (🔧 코드 일치)

| 좌석 | 리클라인(서보) | 회전(스텝) | 슬라이드(스텝) |
| --- | --- | --- | --- |
| 운전석 | ✓ SG90 | ✓ 28BYJ-48/ULN2003 | — |
| 조수석 | ✓ SG90 | ✓ 28BYJ-48/ULN2003 | — |
| 뒷자리1 | ✓ SG90 | — | ✓ NEMA17/TMC2208 |
| 뒷자리2 | ✓ SG90 | — | ✓ NEMA17/TMC2208 |

**🔧 구현 관찰**

- **드라이버 단일화 실패(현 상태)** — 앞좌석 회전은 소형 28BYJ-48/ULN2003(무 STEP/DIR), 뒷좌석 슬라이드는 TMC2208(STEP/DIR/EN). 설계의 "TMC2209 한 종류로 통일"은 이루어지지 않았다. 발표에서 "존별 최적 드라이버 선택"으로 프레이밍하거나, 통일이 목표면 앞좌석 드라이버 교체가 남은 작업.
- **전부 개루프** — 스텝·주행 모두 위치/속도 피드백을 제어에 쓰지 않는다(스텝 펄스 카운트, 주행 open-loop feed-forward + 슬루제한). 구동 엔코더(TIM2/TIM3)는 측정 RPM을 Drive_Status로 보고만 한다.
- **위치 재확정 수단** — 리어 슬라이드에 수동 홈(슬라이드 값 254=하드스톱 탐색) / 재영점(255) 센티넬만 있고, 자동 호밍(`SLIDE_HOMING`)은 기본 비활성(스킵). AS5600 절대 엔코더는 미실장.
- **자기잠금** — 슬라이드 리드스크류로 확보(설계 유지).

## 4. 시스템 아키텍처 (분산 CAN)

### 4.1 구조도 (🔧 실장 4노드 + 설계 감시노드)

```
            [입력]  아우라 레이싱휠 (USB HID)   터치 모니터 (USB 터치)
                          │                          │
                          └────────────┬─────────────┘   ※ 입력 = "요청"일 뿐
                                        ▼
┌──────────────────────────────────────────────────────────────────────┐
│  ① RPi5 + MCP2515 · 통합 슈퍼바이저 노드  (🔧 PySide6+QML 실장)        │
│   - 레이싱휠 읽기(pygame HID: 조향 axis0·페달 axis1·패들 btn4/5)       │
│   - 터치 HMI(모드·좌석) + 실시간 3D 디지털 트윈(QtQuick3D) + 상태·로그  │
│     → 단일 통합 제어 화면(HDMI 1280×720)                              │
│   - 인터록 판단(기어=P 전이잠금 / DRIVE에서만 D/R)                     │
│   - 안무 지휘(좌석별 프리셋 송신) · Drive_Cmd 50ms 주기 송신            │
│   - CAN = MCP2515(SPI) → SocketCAN can0 500kbps                        │
│   - ✗ Heartbeat 송신 없음 · ✗ SafeAbort 송신 없음 (설계 미구현)        │
└──────────────────────────────────────────────────────────────────────┘
        │ CAN 500kbps · 클래식 CAN · 표준 11비트 ID · 멀티마스터 브로드캐스트
        │
        ├── ② [설계] 감시 노드 (STM32)  ※ 🔧 미실장 — 펌웨어 없음
        │      - RPi5 Heartbeat 감시 → SafeAbort. 현재 코드/보드 부재.
        │
        ├── ③ 프론트 존 ECU  🔧 STM32F103CBTx · 베어메탈 슈퍼루프
        │      - 앞 회전2(28BYJ-48/ULN2003) + 앞 리클라인2(SG90)
        │      - RX: SeatCmd 0x110/0x111, GearStatus 0x070, SafeAbort 0x010
        │      - TX: SeatStatus 0x210/0x211  · 기어=P에서만 좌석 적용
        │      - ✗ 앤티핀치 스텁(항상 0) · ✗ 하트비트 · ✗ IWDG
        │
        ├── ④ 리어 존 ECU  🔧 STM32F446RE(NUCLEO) · FreeRTOS(CMSIS v2)
        │      - 뒷 슬라이드2(TMC2208) + 뒷 리클라인2(SG90)
        │      - RX: SeatCmd 0x120/0x121, SafeAbort 0x010
        │      - TX: SeatStatus 0x220/0x221
        │      - ✓ 앤티핀치 INA226(서보 2개, 0x40/0x41, 160mA, 15° 후퇴)
        │      - ✗ 하트비트 · ✗ IWDG
        │
        └── ⑤ 구동 ECU  🔧 STM32F446RE(NUCLEO) · FreeRTOS(CMSIS v1)
               - 주행 바퀴 2륜 차동(JGB37-520/L298N, 개루프)
               - RX: Drive_Cmd 0x100, GearStatus 0x070, SafeAbort 0x010
               - TX: Drive_Status 0x200(측정RPM·기어)
               - ✓ 기어 D/R에서만 구동 · ✓ Drive_Cmd 300ms 손실→정지
               - ✗ mode==DRIVE 로컬체크 없음 · ✗ 하트비트 · ✗ IWDG · 페달 ADC 없음
```

> 🔧 **버스의 실제 라이브 노드 = 4개**(RPi5 + 존 ECU ×3). 설계의 감시 노드(⑤ → 5노드)는 아직 없다.

### 4.2 CAN 버스 (🔧 실측)

- **클래식 CAN 500kbps**, 표준 11비트 ID — 4개 노드 코드 전부 500k로 일치 확인.
- 트랜시버 = STM32 노드는 개별 트랜시버(SN65HVD230 등), RPi5는 MCP2515 모듈 내장 트랜시버.
- 종단 120Ω은 물리 버스 양 끝 2개만.
- CAN 접속 코덱: 슈퍼바이저는 **cantools로 `supervisor/dbc` 로드**, 리어·구동 ECU는 **cantools 생성 `model_car_net.c/.h`** 코덱, 프론트 ECU만 **하드코딩 레이아웃**. (정본 DBC·동기화 이슈는 §4.3)

### 4.3 DBC / CAN 메시지 맵 — 🔧 authoritative (supervisor/dbc/model_car_net.dbc · 최신 2026-06-30)

**정본 = `supervisor/dbc`** (슈퍼바이저 앱 `can_hub.py`가 실제 로드하는 파일, 커밋 `30199d7` "CAN 양방향 연동 + 휠 입력 통합 + 체크섬 ECU 정합"). **노드(6)**: HMI_Controller, Central_Supervisor, Front_Zone_ECU, Rear_Zone_ECU, Drive_ECU, **Monitor_Node**(감시 노드 — DBC엔 정의, 펌웨어 미실장). 총 **14 메시지**, 표준 11비트·리틀엔디안.

| Hex ID | 이름 | DLC | 송신→수신 | 주요 신호 | 🔧 코드 연동 |
| --- | --- | --- | --- | --- | --- |
| **0x010** | `SafeAbort` | 3 | Supervisor→전존 | `Stop_Flag`, `Source_Id`, `Reason_Code` | 존 ECU **수신만**, 송신 없음 |
| **0x050** | `Heartbeat` | 1 | Supervisor→Drive/Monitor | `Alive_Counter` | 송수신 전무 |
| **0x060** | `Heartbeat_Ack` | 1 | **Monitor_Node**→Supervisor | `Ack_Alive_Counter` | 송수신 전무(감시노드 부재) |
| **0x070** | `GearStatus` | 1 | Supervisor→Drive/Front/Rear | `Gear`(0=P/1=D/2=R) | ✅ 슈퍼바이저 TX, 프론트·구동 RX |
| **0x100** | `Drive_Cmd` | 8 | Supervisor→Drive | `Target_Velocity`(0.1RPM), `Steering_Angle`(±135°), `Brake_Depth`(%) | ✅ 50ms 주기 TX (기어는 별도 0x070) |
| **0x110** | `Driver_Seat_Cmd` | 4 | Supervisor→Front | `Drv_Recline`, `Drv_Rotation`, `Rolling_Counter`, `Checksum` | ✅ (롤링+8bit합 체크섬) |
| **0x111** | `Passenger_Seat_Cmd` | 2 | Supervisor→Front | `Psgr_Recline`, `Psgr_Rotation` | ✅ (롤링/체크섬 없음, 비대칭) |
| **0x120** | `Rear_Left_Seat_Cmd` | 2 | Supervisor→Rear | `RL_Recline`, `RL_Slide_Position`(mm) | ✅ |
| **0x121** | `Rear_Right_Seat_Cmd` | 3 | Supervisor→Rear | `RR_Recline`, `RR_Slide_Position`, `Cargo_Lamp_Status`(잔재) | ✅ (램프비트 디코드하나 GPIO 미연결) |
| **0x200** | `Drive_Status` | 8 | Drive→Supervisor | `Current_Velocity`, `Drive_Motor_Current`(🔧 0 고정), `Current_Gear_Status` | ✅ RX |
| **0x210** | `Driver_Seat_Status` | 3 | Front→Supervisor | `Curr_Drv_Recline`, `Curr_Drv_Rotate`, `Drv_Pinch_Detected` | ✅ RX (회전 피드백 포함) |
| **0x211** | `Passenger_Seat_Status` | 3 | Front→Supervisor | `Curr_Psgr_Recline`, `Curr_Psgr_Rotate`, `Psgr_Pinch_Detected` | ✅ RX |
| **0x220** | `Rear_Left_Seat_Status` | 2 | Rear→Supervisor | `Curr_RL_Recline`, `RL_Pinch_Detected` (슬라이드 피드백 없음) | ✅ RX |
| **0x221** | `Rear_Right_Seat_Status` | 2 | Rear→Supervisor | `Curr_RR_Recline`, `RR_Pinch_Detected` | ✅ RX |

> **✅ 기어는 독립 메시지 `GearStatus`(0x070)** — Drive_Cmd 안에 기어 신호가 없다. 슈퍼바이저가 0x070으로 주기 브로드캐스트하고 프론트·구동 ECU가 수신한다. (낡은 interface DBC는 기어를 Drive_Cmd 신호로 갖고 있어 다름 — 아래.)
> **✅ 감시 노드는 CAN 계약(DBC)엔 존재** — Monitor_Node가 `Heartbeat`(0x050) 수신자이자 `Heartbeat_Ack`(0x060) 송신자로 정의돼 있다. 즉 인터페이스는 감시 노드를 전제하나 **펌웨어/보드는 아직 없다**(§6.3).

> **⚠️ 🔧 DBC 2벌 — `interface/dbc`는 낡음, 반드시 동기화**
> 저장소에 DBC가 두 벌 있는데 **최신·정본은 `supervisor/dbc`(2026-06-30, 14메시지, 위 표)**, `interface/dbc`는 **초기 업로드본(2026-06-19 "Add files via upload", 12메시지)으로 오래됐다.** 낡은 `interface/dbc`에서 다른/빠진 점:
> - `GearStatus`(0x070) **없음**(기어가 Drive_Cmd 안의 신호로만 존재)
> - `Heartbeat_Ack`(0x060) **없음**, `Monitor_Node` 노드 정의 없음
> - `SafeAbort`가 `HMI_Emergency` **1바이트**(플래그만)로 되어 있음(정본은 SafeAbort 3바이트)
> - `Driver_Seat_Cmd` **2바이트**(롤링카운터/체크섬 없음)
> - 앞좌석 `*_Seat_Status` **2바이트**(회전 피드백 없음; 정본은 3바이트)
>
> **조치(진행):** 낡은 `interface/dbc/model_car_net.dbc`는 **삭제**했다(2026-07-03). 정본은 `supervisor/dbc/model_car_net.dbc` 단일 관리. 후속으로 cantools 생성 코덱(리어·구동 ECU `model_car_net.*`)을 이 정본에서 재생성하고, 프론트 ECU 하드코딩 레이아웃도 정본(3바이트 Status·4바이트 Driver_Cmd·GearStatus 0x070)에 맞춰 정합. (공유 계약 위치를 `interface/`로 표준화하려면 정본을 그쪽으로 옮겨 관리.)

## 5. 입력 / 제어 설계

### 5.1 터치 HMI = 통합 제어 화면 (🔧 PySide6 + QtQuick3D 실장)

- **모드 선택** — 2×2 타일(`ModeSelect.qml`) → 해당 모드 안무.
- **좌석 개별 제어** — `SeatDetail.qml`에서 좌석 선택 후 축(리클라인/회전/슬라이드) 조그 + Apply.
- **상태 표시 + 실시간 3D 디지털 트윈(신규 강조)** — `Cabin3D.qml`(QtQuick3D)이 4개 좌석 포즈(회전·리클라인·슬라이드)를 실시간 3D로 미러링. 스포츠카 GLB 모델(X-ray 투시·조향 연동 바퀴 회전) + 기어(P/D/R)·인터록·RPM(Drive_Status)·로그를 한 화면에.
  - 화면 상태: **ACTIVE**(좌 40% 3D + 우 60% 패널) ↔ **AMBIENT**(3D 전체화면, 패널 슬라이드아웃). 10초 무입력(기어=P일 때)→AMBIENT.
- 표시 = 포터블 터치 모니터(HDMI 영상 + USB 터치 HID). env 플래그: `HMI_WINDOWED`/`HMI_SELFTEST`/`HMI_NOCAN`.

### 5.2 아우라 레이싱휠 (🔧 USB-HID 경로 확정 — 설계 "길 A")

**하드웨어** — 겜맥 아우라(USB HID, 2페달). **RPi5가 `pygame.joystick`으로 직접 읽는다**(evdev는 아날로그 페달 깊이 손실로 폐기, `SDL_VIDEODRIVER=dummy`).

- **매핑(jstest 검증)**: 조향=`axis 0`(±1→×127, 데드밴드 3°), 페달=`axis 1` **단일 축**(음수=가속, 양수=브레이크), 기어업 패들=`button 5`, 기어다운=`button 4`(라이징엣지 디바운스).
- **기어**: 휠이 기어 상태를 소유하지 않음. 패들 엣지 → `gearShift(±1)` → `GEARS=("R","P","D")` 스텝 → 터치 슬라이더와 **동일한 인터록 경로**(`_apply_gear`).
- **디바이스 선택(중요 제약)**: `_select_wheel`이 터치패널(`usb2iic`,`ctp`,`touch`)·IMU를 이름으로 제외하고 **축=6개인 P4X("Generic X-Box pad", D-INPUT) 모드**를 선호. 축≠6이면 페달 매핑이 깨지므로 경고 로그. → **휠은 P4X 6축 모드여야 페달·기어가 정상**(스캔 로그 `wheel_scan.log`).
- **드라이브-바이-와이어**: throttle% → RPM(`RPM_PER_PCT=3`, 상한 300) → `Drive_Cmd` **50ms 주기 송신**(P에서는 중립 0,0,0).

> 🔧 **설계 "길 B"(STM32 ADC APPS)는 채택 안 됨.** 구동 ECU에 ADC 코드 자체가 없다(비활성). 페달 필터체인(RC/오버샘플/median/EMA)·APPS plausibility는 현재 미구현 — 필요하면 슈퍼바이저 쪽 `wheel_input.py`에 소프트 필터를 얹는 형태가 현실적.

### 5.3 요청 → 승인 → 실행 흐름 (🔧 구현)

```
[HMI] 모드 선택 / 좌석 조그 ─┐
[휠]  기어/조향/페달        ─┤→ 슈퍼바이저 인터록(기어=P? DRIVE?) 
                            │   → 통과: CAN Seat_Cmd / Drive_Cmd 송신
                            │   → 거부: modeLocked/gearLocked UI 표시
[구동 ECU] Drive_Cmd 300ms 손실 → 자체 정지 (유일한 링크 페일세이프)
```

### 5.4 기어/상태 — 브로드캐스트 (🔧 부분)

- `GearStatus`(0x070)는 주기 브로드캐스트, 값 안 바뀌어도 송신 → 구동 ECU가 최신값 보유. (🔧 슈퍼바이저 `send_gear_status` 구현)
- 실시간 상태(조향·페달)는 `Drive_Cmd`(0x100)에 묶여 50ms로 나감.
- 📐 페일세이프(GearStatus 끊기면 P 폴백)는 설계 항목 — 🔧 구동 ECU는 Drive_Cmd 손실 타임아웃(300ms)으로 대체 커버.

## 6. 안전 설계 — 📐 설계 vs 🔧 구현 현황

> **핵심 메시지:** "정상 파이프라인 인터록"은 상당수 구현, "고장 대응 안전망(생존감시·워치독)"은 대부분 미구현. 발표 시 이 구분을 명확히.

### 6.1 인터록 판단 위치 + 구현 상태

| 인터록 | 판단 위치 | 📐 목표 | 🔧 구현 |
| --- | --- | --- | --- |
| 주행 금지(기어≠D/R → 바퀴 차단) | 구동 STM32 | 로컬 즉시 | ✅ 구현(기어만; mode==DRIVE 로컬체크는 없음) |
| 끼임 정지(과전류) | 해당 STM32 | 로컬 즉시 | ◐ **리어만**(INA226 서보 2개). 프론트는 스텁(항상 0), 구동은 대상 아님 |
| 스윙 경로 클리어런스 | 슈퍼바이저 | 전역 | ✗ 미구현(순차 안무로 간접 회피) |
| 모드 전이 사전조건(기어=P) | 슈퍼바이저 | 전역 | ✅ 구현(`selectMode`/`_seat_cmd_approved`) |
| 링크 페일세이프(타임아웃) | 존 ECU | ~수백ms | ◐ **구동만** Drive_Cmd 300ms 손실 정지. 프론트·리어 없음 |
| IWDG 하드웨어 워치독 | 존 ECU | ~100ms–1s | ✗ **전 노드 미구현**(HAL_IWDG 비활성) |

### 6.2 E-stop / SafeAbort (CAN 0x010) — 🔧 수신만 구현

- **수신**: 프론트(`FrontSeat_StopAll`)·리어(`slide_estop`)·구동(`is_emergency_active`) 3개 ECU 모두 0x010 stop_flag로 즉시 정지 구현. 출처불문 수용(설계 준수).
- **송신**: 🔧 **없음.** 슈퍼바이저(RPi5)에 SafeAbort 송신 코드가 없다. 물리 E-stop·감시노드도 미실장. 현재 발동 수단은 **테스트 스크립트**(`cantest.py`의 `e/r` 키)뿐.
- 리어 E-stop은 스텝만 정지(서보는 강제 유지 안 함) — 보완 여지.

### 6.3 하트비트 / 감시 노드 / split-brain 방지 — 🔧 미구현(설계 단계)

- 📐 하트비트 최우선 ID + 다회 누락 타임아웃 + 독립 감시 노드 + 명령/안전 비대칭. → 🔧 **Heartbeat(0x050) 메시지를 어떤 노드도 송수신하지 않음.** 감시 노드 펌웨어·보드 부재. 명령/안전 비대칭은 "동작명령=슈퍼바이저만 송신"이 사실상 성립(감시노드가 없어 발신자가 하나)하나, 설계가 의도한 이중 안전망은 아직 없다.
- **현재 유일한 생존성 메커니즘** = 구동 ECU의 Drive_Cmd 50ms 스트림 + 300ms 손실 정지.

### 6.4 좌석 과전류 긴급정지(seatEstop) — 🔧 신규 UI, CAN 미배선

- vehicle_state에 `seatEstop`/`anyEstop` 프로퍼티 + `setSeatEstop(seat,on)` 슬롯, `Seat3D.qml`에 빨강↔핑크 무한 깜빡임(끼임·이동보다 우선) 추가.
- **단, CAN 수신 미배선** — 코드 주석이 "CAN 수신 방식 미정"이라 명시. 현재는 `setSeatEstop` 슬롯(테스트/후속 훅)으로만 세팅. 후속으로 과전류 비트 또는 SafeAbort Source_Id를 이 슬롯에 연결해야 실동작.

### 6.5 위치 복구 / FSM — 🔧 부분/미구현

- 📐 "위치 미확정 시 이동 금지" FSM(IDLE/MOVING/LOCKED/FAULT). → 🔧 **어느 ECU에도 명시적 FSM enum 없음.** 이동 여부는 `pos≠target` 암묵 판정, 래치 플래그(estop/pinch)만 존재.
- 위치 재확정: 리어 수동 홈(254)/재영점(255)만. 자동 호밍 비활성, AS5600 없음. 자동복귀 금지 원칙은 estop 래치(리셋 전까지 정지 유지)로 부분 충족.

### 6.6 인증 주장 금지 — "안전 지향 설계 패턴"으로만 프레이밍. (특히 미구현 항목을 "구현했다"고 말하지 않는다.)

## 7. 펌웨어 설계 (존 ECU) — 🔧 실측

### 7.1 실행 모델 (존별 상이 — 설계의 "공통 FreeRTOS 뼈대" 미달성)

| 존 | MCU | 실행 모델 | 태스크 |
| --- | --- | --- | --- |
| 프론트 | STM32F103CBTx | **베어메탈 슈퍼루프**(72MHz) | while(1): CanEvents→Step_Process→StatusIfDue |
| 리어(slide4) | STM32F446RE | **FreeRTOS CMSIS v2**(84MHz) | MotionTask(High)·StatusTask(100ms)·defaultTask, CAN RX 큐 |
| 구동 | STM32F446RE | **FreeRTOS CMSIS v1**(180MHz) | StartDriveTask(10ms)·StartCanTxTask(10ms) |

> 🔧 설계의 4태스크 표준 뼈대(CAN_Rx/MotorCtrl/Safety/Status_Tx)는 어느 노드에도 그대로 구현되지 않았다. 프론트는 RTOS 자체가 없다. 통일이 목표면 프론트 F103 RTOS 포팅이 남은 작업(단, 슈퍼루프로도 기능은 동작 중).

### 7.2 FSM / 정적분석

- 🔧 계층형 FSM(System Mode / 좌석 서브 FSM) enum은 **미구현**. 모드 전이 안무는 슈퍼바이저가 좌석별 프리셋 송신으로 수행.
- 🔧 리어/구동은 cantools 생성 DBC 코덱 사용, 프론트는 하드코딩. MISRA `cppcheck` 정적분석은 문서 계획 항목(코드 내 증거 없음).

### 7.3 정리 필요한 중복/잔재 (🔧)

- **ecu-rear-cargo**: `firmware/slide2`(L298N 단일축, 구식)·`slide3`(TMC2208 슈퍼루프)·`slide4`(canonical FreeRTOS) 3벌 병존 → slide2/slide3 정리 권장.

### 10.2 남은 우선순위 (🔧 기준 재정렬)

1. **DBC 단일화**(interface/ 기준 재생성 + GearStatus/SafeAbort/체크섬 반영 + 전 노드 코덱 갱신) — 통합 리스크 1순위.

6. 중복 프로젝트(slide2/3, PowerTrain 베어메탈) 정리 + cargo 잔재 정리.
